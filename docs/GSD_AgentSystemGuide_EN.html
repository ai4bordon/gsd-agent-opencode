GSD (Get Shit Done): complete guide to the agent system

This guide describes full GSD operation in OpenCode: all slash commands, all agents, all key pipeline scenarios, automatic subagent invocations, and artifacts in .planning/.

1. What this is and how to read it

GSD is a command-first agentic development system. Core principle: the user runs /gsd-*, and the command-level orchestrator launches specialized agents via Task(subagent_type=...) where needed.

  • If you need a fast start: begin with Pipeline variants.
  • If you need a full reference: use All commands and All agents.
  • If you need internals: see Automatic chains and Artifacts.

2. System architecture

User
  -> /gsd-* command (commands/gsd/*.md)
  -> step orchestration + .planning read/write
  -> subagent execution (agents/*.md) when needed
  -> artifact updates (PLAN/SUMMARY/VERIFICATION/UAT/STATE/ROADMAP)
  -> next-command guidance

4.1 Detailed role of the main orchestrator Gsd-agent

Gsd-agent is the central orchestrator that manages not a single answer, but a continuous delivery flow: idea -> context -> plan -> execute -> verify -> close.

  • Pipeline dispatcher: selects the next valid /gsd-* command and keeps sequence integrity.
  • Command-centric mode: explicit slash commands take priority over ad-hoc manual orchestration.
  • Source of truth: relies on .planning/STATE.md, .planning/ROADMAP.md, .planning/REQUIREMENTS.md for routing and continuity.
  • Continuation policy: if Next command: /gsd-... was issued, replies like "ok/continue/yes" must execute that pending command literally.
  • Priority rules: user explicit command > pending command > generic autonomy.
  • Quality control: does not allow skipping verify/gap-loop when gaps exist; routes to /gsd-plan-phase --gaps and /gsd-execute-phase --gaps-only.
  • Autonomy boundaries: executes safe reversible actions autonomously, asks confirmation for irreversible/high-risk actions (prod/security/billing/secrets).
  • Expected step output: after each stage, provides what was done, what was found, and one concrete next command.

Mini decision tree:

Step start
  |
  |-- Did user provide an explicit /gsd-* command?
  |      |-- Yes -> Execute exactly that command
  |      |-- No
  |
  |-- Is there a pending "Next command: /gsd-..." and user said "ok/continue/yes"?
  |      |-- Yes -> Execute pending command literally
  |      |-- No
  |
  |-- Does .planning/ exist?
  |      |-- No -> /gsd-new-project (or /gsd-map-codebase -> /gsd-new-project)
  |      |-- Yes
  |
  |-- Is current status clear from STATE/ROADMAP?
  |      |-- No -> /gsd-progress
  |      |-- Yes -> choose next pipeline stage:
  |                discuss -> plan -> execute -> verify -> close
  |
  '-- If gaps found -> mandatory gap-loop:
         /gsd-plan-phase N --gaps -> /gsd-execute-phase N --gaps-only

Practical outcome: the user always knows where they are in the process, what is already done, and what exact command comes next without manual stitching.

5. Automatic chains (command -> orchestration -> subagents -> artifacts)

  1. /gsd-new-project: preflight and questioning -> optional research (4 x gsd-project-researcher) -> gsd-research-synthesizer -> requirements -> gsd-roadmapper -> PROJECT/REQUIREMENTS/ROADMAP/STATE/config.
  2. /gsd-plan-phase N: phase and mode selection -> optional gsd-phase-researcher -> gsd-planner -> gsd-plan-checker -> up to 3 planner/checker revisions -> *-PLAN.md.
  3. /gsd-execute-phase N: read all plan files -> group by wave -> parallel gsd-executor workers per wave -> collect summaries -> gsd-verifier -> *-SUMMARY.md + *-VERIFICATION.md + roadmap/state updates.
  4. /gsd-verify-work N: conversational UAT by deliverables -> write *-UAT.md -> if issues: diagnose + gap planning via gsd-planner and gsd-plan-checker -> prepare /gsd-execute-phase N --gaps-only.
  5. /gsd-audit-milestone: aggregate milestone verification/summaries -> run gsd-integration-checker -> final MILESTONE-AUDIT.md -> route to completion or gap-closure.
  6. /gsd-quick: small task intake -> gsd-planner (quick mode) -> gsd-executor -> separate quick artifacts + state update.

5.1 Built-in loops and how RALF Lite is used

What counts as built-in loops in the system:

  • Planner -> Checker loop (up to 3 iterations): after gsd-planner, gsd-plan-checker runs; if ISSUES FOUND, plan is revised, up to 3 rounds.
  • Phase gap-loop: if gaps exist, mandatory pair /gsd-plan-phase N --gaps -> /gsd-execute-phase N --gaps-only until acceptable result.
  • Post-execution verify-loop: after gap plan re-execution, phase verification runs again until passed.
  • Wave-loop in execute-phase: plans run in waves; retries inside a wave target only failed autonomous plans.
  • Checkpoint continuation loop: at checkpoint:*, agent stops, orchestrator collects user decision, then spawns a fresh continuation agent.
  • UAT -> diagnosis -> fix planning loop: UAT issues are diagnosed, then fix/gap plans are generated and re-executed.
  • Debug investigation loop: gsd-debugger follows evidence -> hypothesis -> test -> evaluate.
  • Settings menu loop: gsd-settings menu repeats until explicit exit.

How RALF Lite is used:

  • Purpose: bounded retry-and-repair loop only where outcomes are objectively verifiable.
  • Where it applies: in execute-plan for type="auto" tasks, and in execute-phase for autonomous: true plans.
  • Where it does not apply: it does not bypass checkpoint:* and does not replace architecture decisions requiring human choice.
  • Stop conditions: success, reaching max_iterations, repeated error fingerprint, no progress.
  • Reporting: final summaries should include RALF Lite Loop Report with attempts and stop reason.
  • How to disable globally: edit ${OPENCODE_CONFIG_DIR:-~/.config/opencode}/get-shit-done/config.json and set ralf_lite.enabled: false (or disable only apply_to.auto_tasks/autonomous_plans).
  • How to override per project: add ralf_lite section in .planning/config.json with needed fields (enabled, max_iterations, error_fingerprint_repeats, no_progress_repeats, apply_to.*).
  • Important config note: /gsd-settings does not edit RALF Lite defaults; edit global config or project override manually.

6.1 Greenfield (new project)

/gsd-new-project
/gsd-discuss-phase 1
/gsd-plan-phase 1
/gsd-execute-phase 1
/gsd-verify-work 1
# if gaps exist
/gsd-plan-phase 1 --gaps
/gsd-execute-phase 1 --gaps-only
# then next phase or milestone audit

6.2 Brownfield (existing code)

/gsd-map-codebase
/gsd-new-project
/gsd-progress
/gsd-discuss-phase 1
/gsd-plan-phase 1

6.3 Phase cycle from context to acceptance

/gsd-discuss-phase N
/gsd-research-phase N     # optional
/gsd-plan-phase N
/gsd-execute-phase N
/gsd-verify-work N

6.4 Gap-closure cycle (when goal is not yet achieved)

/gsd-plan-phase N --gaps
/gsd-execute-phase N --gaps-only
/gsd-verify-work N         # or /gsd-audit-milestone at milestone end

6.5 Milestone close cycle

/gsd-audit-milestone [version]
/gsd-plan-milestone-gaps   # if audit found gaps
/gsd-complete-milestone X.Y.Z
/gsd-new-milestone "Next scope"

6.6 Incident/debug cycle

/gsd-debug "issue symptom"
# if needed after /new
/gsd-debug
# then fix via quick or phase pipeline

9. Nuances, rules, and edge cases

9.1 Critical quality protocols

  • Plan quality loop: after gsd-planner, gsd-plan-checker runs; if issues remain, up to 3 revision rounds.
  • Goal-backward verify: gsd-verifier validates actual code delivery, not just report claims.
  • RALF Lite: bounded auto-retry loops (usually up to 3) that stop on repeated fingerprint or no progress.
  • Checkpoints: human-verify, decision, human-action from references/checkpoints.md.
  • Gap loop is mandatory: gaps from verify/UAT/audit are never ignored; they are closed via --gaps and/or milestone gap phases.

9.2 Git and planning document behavior

  • Behavior is controlled by planning.commit_docs and planning.search_gitignored (see references/planning-config.md).
  • With commit_docs=false, planning artifacts can stay local and be excluded through .gitignore.
  • Executor commits are atomic; broad staging such as git add . inside task commits is prohibited.

9.3 Detected inconsistencies across files

  • Milestone audit path: commands/gsd/audit-milestone.md contains .planning/v{version}-v{version}-MILESTONE-AUDIT.md, but other files use .planning/v{version}-MILESTONE-AUDIT.md (including commands/gsd/plan-milestone-gaps.md, commands/gsd/complete-milestone.md).
  • Trigger docs vs actual calls: agents/gsd-roadmapper.md and agents/gsd-research-synthesizer.md document only /gsd-new-project, but they are also invoked in commands/gsd/new-milestone.md.
  • Completion route: commands/gsd/progress.md may suggest /gsd-complete-milestone without version, while commands/gsd/complete-milestone.md requires mandatory <version>.
  • Agent count in mapping: agents/gsd-set-profile.md and agents/gsd-set-model.md mention "11 agents", but the mapped list is larger.
